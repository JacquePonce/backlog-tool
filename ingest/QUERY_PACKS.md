### 2a. Daily Slack pull for `daily_refresh` (token-boundary channel filter + DMs)

Used by [`scripts/sources/slack.py`](../scripts/sources/slack.py). Agent writes two JSON files under `ingest/`:

- `last_slack_mentions.json` — one entry per message where the user is `@mentioned` in a channel whose name matches the rule below.
- `last_slack_dms.json` — one entry per DM / MPIM thread with unread activity since the watermark.

**Channel name match rule (token boundary):**

Split the channel name on `-`; keep the channel iff its token set intersects `{troy, us, da, ccf}`.

| Channel | Tokens | Included |
|---------|--------|----------|
| `troy-cc-beta` | troy, cc, beta | yes (troy) |
| `us-credit-strategy` | us, credit, strategy | yes (us) |
| `da-task-force` | da, task, force | yes (da) |
| `ccf-core` | ccf, core | yes (ccf) |
| `business-ops` | business, ops | no |
| `status-updates` | status, updates | no |

**Channel mentions (output: `last_slack_mentions.json`)**

1. `conversations_list types=public_channel,private_channel` → filter by token-boundary rule above; collect `channel_id`, `channel_name`.
2. For each matching channel, `conversations_history oldest=<sync_state.slack_mentions.last_run>` (epoch).
3. Keep messages whose text contains `<@U...>` mention of the user (or the literal `@jacqueline.ponce`).
4. Get `chat_getPermalink` per message. Emit one row per message:

```json
{"messages": [{
  "channel_id": "C07...",
  "channel_name": "troy-cc-beta",
  "ts": "1772641511.021559",
  "thread_ts": "1772641500.000000",
  "permalink": "https://nubank.slack.com/archives/C07.../p1772641511021559",
  "user": "U02...",
  "user_name": "Vitoria Previato",
  "text": "full message text",
  "text_snippet": "first ~200 chars"
}]}
```

**DMs and MPIMs (output: `last_slack_dms.json`)**

1. `conversations_list types=im,mpim` for user.
2. For each, `conversations_history oldest=<sync_state.slack_dms.last_run>`.
3. Skip threads whose latest activity is from the user (`last_author_is_me = true`).
4. Emit one row per thread:

```json
{"threads": [{
  "channel_id": "D04...",
  "channel_type": "im",
  "participants": ["Pedro Silva"],
  "thread_ts": "1772641000.000000",
  "latest_ts": "1772641511.021559",
  "permalink": "https://nubank.slack.com/archives/D04.../p1772641511021559",
  "last_author": "Pedro Silva",
  "last_author_is_me": false,
  "preview": "first ~200 chars of the newest unread message"
}]}
```

**Stable ids generated by the driver**: `slack-<channel_id>-<ts>` for channel mentions, `slack-dm-<channel_id>-<thread_ts or latest_ts>` for DMs.

## 7. Google Calendar + Gemini transcriptions

Used by [`scripts/sources/gcal_gemini.py`](../scripts/sources/gcal_gemini.py). Two JSON files:

### 7a. Events (`last_gcal_events.json`)

1. `calendar_listEvents` on the primary calendar for window `[sync_state.gcal_gemini.last_run - 1d, now + 1d]` (defaults to today if no watermark).
2. Keep events where the user appears in `attendees` (any `responseStatus`, including `needsAction`).
3. For each event, capture any attached Gemini notes URL from:
   - `event.description` (search for `docs.google.com/document/d/...`)
   - `event.attachments[]`
   - Drive fallback: `drive_search` with `name contains '<event.summary>' and mimeType='application/vnd.google-apps.document'` in `Meet Recordings` folder.

```json
{"events": [{
  "id": "evt_abc",
  "summary": "Troy CC Beta sync",
  "start": "2026-04-21T14:00:00-03:00",
  "end":   "2026-04-21T15:00:00-03:00",
  "attendees": ["jacqueline.ponce@nubank.com.br", "..."],
  "hangoutLink": "https://meet.google.com/...",
  "notes_doc_url": "https://docs.google.com/document/d/..."
}]}
```

### 7b. Parsed transcription action items (`last_gemini_docs.json`)

For each event with `notes_doc_url`, `docs_read` the doc and extract items for the user under these section headers (case-insensitive):

- English: `Action items`, `Next steps`, `Follow-ups`, `TODO`
- Portuguese: `Próximos passos`, `Ações`, `Ação`, `Pendências`, `Follow-up`, `A fazer`

Keep a bullet iff **any** is true:

- It starts with (or contains) a name match: `Jacqueline`, `Jacque`, `Jacqueline Ponce`, `@jacqueline.ponce`.
- It appears under a sub-heading like `Jacqueline:` / `Assigned to Jacqueline` / `Responsável: Jacqueline`.
- The doc only has one assignee set and it is the user.

Emit:

```json
{"docs": [{
  "event_id": "evt_abc",
  "doc_id": "1AbCdEf...",
  "url": "https://docs.google.com/document/d/.../edit",
  "title": "Notes by Gemini - Troy CC Beta sync - 2026-04-21",
  "fetched_at": "2026-04-21T16:05:00Z",
  "action_items": [
    {"text": "Confirm sponsor bank next step with Pedro", "assignee_hint": "Jacqueline Ponce", "language": "en"},
    {"text": "Enviar resumo da call para o squad", "assignee_hint": "Jacqueline", "language": "pt"}
  ]
}]}
```

**Stable id generated by the driver**: `gmeet-<event_id>-<index>` (index within `action_items`).

## 8. Confluence comments (for replies)

Used by [`scripts/sources/confluence.py`](../scripts/sources/confluence.py). Output: `last_confluence_comments.json`.

1. Atlassian MCP: search pages with CQL `creator = currentUser() OR mention = currentUser()` (last 365 days is fine on a daily cadence; the driver dedupes).
2. For each page, list **inline + page comments** (`getConfluencePageFooterComments` and `getConfluencePageInlineComments` or equivalent MCP call).
3. Keep comments where:
   - The comment body `@mentions` the user, OR
   - The page was **authored by the user** AND the comment has **no later reply from the user** in the same thread (heuristic: scan thread replies; `has_my_reply_after = false`).
4. Skip comments whose author is the user.

Emit:

```json
{"comments": [{
  "page_id": "123456789",
  "page_title": "Troy CC Beta — Monitoring Hub",
  "page_url": "https://nubank.atlassian.net/wiki/spaces/CR/pages/123456789",
  "page_author_is_me": true,
  "comment_id": "987654",
  "comment_author": "Pedro Silva",
  "comment_author_is_me": false,
  "mentions_me": false,
  "has_my_reply_after": false,
  "text": "Pode revisar a seção de roll rates?",
  "created_at": "2026-04-20T18:22:00Z",
  "url": "https://nubank.atlassian.net/wiki/spaces/CR/pages/123456789?focusedCommentId=987654"
}]}
```

**Stable id generated by the driver**: `cfl-<page_id>-<comment_id>`.

## 3. Jira (when Atlassian MCP authenticated)

**Source of truth for JQL** used before saving to [`last_jira_currentUser.json`](last_jira_currentUser.json): this section. [`scripts/refresh_from_jira.py`](../scripts/refresh_from_jira.py) only reads the JSON; it does not call Jira.

### “On me” since April 2025

Use **`2025-04-01`** as the start of April 2025 (change the date if you need a different cut).

**Broader — anything assigned to you that changed on or after that date:**

```text
assignee = currentUser() AND updated >= "2025-04-01" ORDER BY updated DESC
```

**Narrower — only issues created on or after that date:**

```text
assignee = currentUser() AND created >= "2025-04-01" ORDER BY updated DESC
```

**Optional — non-done only** (smaller result set; not required for “everything”):

```text
assignee = currentUser() AND updated >= "2025-04-01" AND statusCategory != Done ORDER BY updated DESC
```

### Pagination and large exports

A single `searchJiraIssuesUsingJql` call may cap results (e.g. **100**). If you need **all** issues matching the JQL:

1. Re-run the search with **`startAt`**: `0`, then `100`, then `200`, … until a page returns fewer issues than `maxResults`.
2. **Merge** every response’s **`issues`** arrays into **one** object: `{ "issues": [ ... ] }` (same shape as a single API response).
3. Save that merged JSON as **`ingest/last_jira_currentUser.json`**.

If your MCP allows a higher **`maxResults`**, you can raise it instead of paging—subject to Jira/API limits.

### Daily Jira pull for `daily_refresh`

Used by [`scripts/sources/jira.py`](../scripts/sources/jira.py). Agent writes to `ingest/last_jira_currentUser.json` (same shape as today).

Expand the scope so **reporter = me** is also covered (you asked for both assigned to me **and** created by me):

```text
(assignee = currentUser() OR reporter = currentUser()) AND updated >= "<sync_state.jira.last_run or 2025-04-01>" ORDER BY updated DESC
```

If `sync_state.jira.last_run` is null, fall back to `2025-04-01`. Paginate with `startAt` until a page returns fewer issues than `maxResults`, then merge `issues` arrays.

### Other JQL patterns (no date filter)

```text
assignee = currentUser() ORDER BY updated DESC
```

```text
text ~ "Jacqueline Ponce" OR text ~ "Jacque Ponce" ORDER BY updated DESC
```

```text
labels in (troy-cc-beta, sponsor-bank, blocked-da, DA-task-force) ORDER BY updated DESC
```

### Post-merge filter in `refresh_from_jira.py` (optional to know)

[`refresh_from_jira.py`](../scripts/refresh_from_jira.py) **drops** Jira issues that map to **`front: other`** **and** board **`status: done`** (`jira_issue_included_in_backlog`). Themed fronts still keep done rows. To include **every** `other+done` issue, that filter would need a code change (e.g. a CLI flag)—not applied by default.

Map Jira status → local `status`:

| Jira category | items.yaml `status` |
|---------------|---------------------|
| To Do, Backlog, Selected for Development, Discovery | `backlog` |
| In Progress, In Review | `in_progress` |
| Blocked | `blocked` |
| Done, Closed, Cancelled (if truly complete) | `done` |
